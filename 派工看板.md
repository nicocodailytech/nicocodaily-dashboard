**⚠️ COO 6/15 補充需求（很重要）**：

**🎯 優先順序：SQLite 紀錄庫 > TG 推送**
- SQLite 紀錄是 **主要產出**（資料資產、未來分析基底）
- TG 推送只是 **次要 wrapper**（即時通知）
- 抓資料抓不全、紀錄不完整 = W1 失敗
- 推送可以晚做、紀錄不能少做

**紀錄庫要求（不可妥協）**：
1. `activities` 表 append-only，**永不 UPDATE 永不 DELETE** — 原始事件永久保留
2. `daily_snapshot` 表每天每個 entity 1 列 + **當下情境**（spend / orders / revenue / ROAS / CTR / CPC、含過去 7 天 trend）
3. 紀錄完整度高於即時性 — 寧可 batch 抓慢一點、別漏資料
4. **為未來分析預留 query view**：
   - `SELECT * FROM activities WHERE event_type='paused' AND object_id IN (高 ROAS campaigns)` — 投手停爆款的 pattern
   - `SELECT * FROM daily_snapshot WHERE roas < 1.0 AND status='ACTIVE' ORDER BY date` — 持續燒錢卻沒停的 campaign
   - 預留欄位給 Behavioral Cloning 用

**TG 推送規則**：
- **任何異動都要推**，不要按嚴重度過濾
- 規格書原本設計 3 級分類（L1 立即 / L2 4 小時 / L3 每日摘要）→ **改成全部即時推**
- 嚴重度只用 emoji 區分視覺（🔴🟢🟡📊），但「推送與否」不過濾
- 推送內容必須帶「**當下情境**」：訊息要從紀錄庫 join 最新 daily_snapshot 拿 ROAS / spend 給 COO 判讀
- 訊息格式範例：`🔴 [暫停] 主帳號 14:23 / campaign: XXX / 7d ROAS: 8.4 / by: 投手名字`
